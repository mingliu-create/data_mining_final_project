## 1. 背景與動機

日本三大都市圈（東京、大阪、名古屋）集中了全國 51.8% 的人口，東京圈每年淨遷入人數仍達 11.4 萬人（2023 年），極端的都市化帶來三大核心問題：

- **通勤尖峰交通癱瘓**：密集的早晚通勤人潮超出交通系統負載。
- **突發事件疏散困難**：首都直下型地震預估將產生約 517 萬名歸宅困難者，人流預測可協助避免車站群聚踩踏並優化避難所資源配置。
- **觀光熱點乘載超載（Overtourism）**：京都等地遊客數創歷史新高，需要即時路線導引以平衡居民生活與觀光品質。

日本位置情報（LBS）相關市場規模預測至 2026 年將達 3,500 億日圓，城市人流預測同時具備防災、零售、公共交通彈性調度等實際應用價值。

日本智慧型手機普及率已達 94.2%（2023 年），超過 1 億台活躍裝置搭配密集的 4G/5G 基地台與公共 Wi-Fi，持續產生海量去識別化位置足跡，為訓練人流預測模型提供充足的資料基礎。

## 2. 研究問題

| # | 問題 | 說明 |
|---|------|------|
| Q1 | 城市移動熱點如何分布？ | 在無預設分群的前提下，從原始軌跡資料中識別具有意義的城市移動聚集區域。 |
| Q2 | 不同地點的語意角色為何？ | 車站、學校、商店街等地點在工作日與假日呈現截然不同的人流特性，從資料中萃取這些語意標籤。 |
| Q3 | 個人移動軌跡是否可被預測與生成？ | 整合空間、語意與時序規律性特徵後，建立可依條件生成多樣化個人人流資料的生成模型。 |

## 3. 核心目標

以大規模去識別化移動軌跡資料為基礎，結合資料探勘技術與深度學習模型，分析城市人流的時空規律，並實現個人化移動軌跡的**預測**與**生成**。

整體流程分為兩個階段：

```
原始軌跡資料
    │
    ├── PHASE 1：特徵探勘
    │       ├── HDBSCAN（高密度空間分群，識別移動熱點）
    │       ├── POI 分析（萃取地點語意，理解軌跡目的）
    │       └── 軌跡穩定性（個人歷史分析，量化移動規律）
    │
    └── PHASE 2：深度模型
            └── Conditional VAE
                    ├── 輸入：空間 / 語意 / 規律性特徵
                    ├── 輸出：高精度預測
                    └── 輸出：軌跡資料生成
```

## 5. 資料前處理

### 5.1 空間處理

1. **人流地圖生成**：將每格的唯一使用者數（unique users per cell）繪製成白底紅色地圖，道路輪廓自然浮現，供視覺疊圖使用（輸出：`output/figures/unique_users_map.png`）。
2. **座標系統確認**（已完成）：網格座標與真實地理的對應關係已確認：
   - `x`（0→199）：南→北，對應緯度 34.499°N → 35.505°N
   - `y`（0→199）：西→東，對應經度 136.502°E → 137.499°E
   - 邊界框：SW=(34.4990°N, 136.5021°E)，NE=(35.5047°N, 137.4986°E)
3. **座標對映表**（已完成）：全部 40,000 格的中心點與邊界框存於 `data/grid_to_latlon.csv`，供 POI 查詢使用。

### 5.2 時間處理

1. **平假日判斷**：使用 K-means 對每日人流量進行分群，初步區分工作日與假日；再與日本國定假日曆及特殊事件（颱風、大型展覽）比對，確認確切日期標籤。
2. **時間離散化**：將一天切成 **48 個時階段**（每段 30 分鐘），在保留日內細節的同時去除冗餘資訊，作為後續模型的時序索引。

## 6. 資料探勘方法

### 6.1 HDBSCAN（Hierarchical Density-Based Spatial Clustering of Applications with Noise）

**目標**：識別城市移動熱點（Q1）

選用 HDBSCAN 而非 K-Means 或 DBSCAN 的原因：
- K-Means：需預先指定 K 值，且假設資料為球狀分佈，不符合真實人流樣態。
- DBSCAN：全局固定密度，無法同時處理「擁擠車站」與「稀疏郊區」等密度差異極大的區域。
- **HDBSCAN 優點**：無須給定 K 值、自適應多變密度、高度抗雜訊過濾。

**四步演算法**：

1. **相互可達距離**：計算點對點距離，將低密度雜訊推遠，凸顯真實熱點。
2. **最小生成樹（MST）**：以距離為權重，建構連通所有點且總距離最短的樹狀圖。
3. **層次聚類（Hierarchy）**：依序砍斷最長連線，形成層次式樹狀圖（Dendrogram）。
4. **萃取穩定群集**：計算分支「穩定度（Stability）」，自動選出最抗雜訊熱點。

### 6.2 POI 分析（Point of Interest）

**目標**：識別地點語意角色（Q2）

工作日與假日的人流分布相異（學校在工作日人流較多、商店街在假日人流較多）。此步驟利用 OpenStreetMap Overpass API 抓取名古屋範圍內的 POI 資料，結合每個網格的人流時序向量，對網格進行語意分類，萃取「車站」、「學校」、「商業區」等語意標籤。

**POI 資料取得流程**：

1. 透過空間處理（5.1 節）取得 200×200 網格的真實經緯度邊界框。
2. 以 Overpass API 查詢指定邊界框內的 POI，關注類型包含：
   - `amenity=station`（車站）
   - `amenity=school`（學校）
   - `shop=*`（商店）
   - `leisure=*`（休閒設施）
   - `amenity=restaurant`、`amenity=hospital` 等
3. 將每筆 POI 的座標對齊至最近的 500m 網格，統計每格各類 POI 的數量，形成語意特徵向量。

### 6.3 個人軌跡穩定性分析

**目標**：量化個人移動規律，作為生成模型的條件特徵（Q2、Q3）

步驟：
1. 針對特定時間段（如上下班時間）與特定語意地點（學校、車站、商店街），對使用者進行分群。
2. 將每群使用者依照時序繪製移動軌跡。
3. 分析個人歷史軌跡的重複性，量化其移動規律（如：每天早上 8 點都出現在相同網格）。

## 7. 模型訓練：Conditional VAE（CVAE）

**目標**：預測並生成個人移動軌跡（Q3）

### 7.1 模型架構

採用**條件變分自編碼器（Conditional Variational Autoencoder）**，融合空間、語意與規律性特徵進行條件式訓練。

```
Input x（原始軌跡序列）
    + Label y（個人軌跡特徵：HDBSCAN 群集 ID、POI 語意標籤、其他規律性特徵）
        │
        ▼
    Encoder q_φ(z|x, y)
        │ ── 輸出 μ_φ, σ_φ
        ▼
    Latent Space z（取樣自多變量分佈）+ Conditioning Label y
        │
        ▼
    Decoder p_θ(x|z, y)
        │
        ▼
    Reconstructed / Generated Trajectory x̂
```

### 7.2 訓練條件（Label y）

- **空間特徵**：HDBSCAN 分群結果（使用者的主要活動熱點 ID）
- **語意特徵**：POI 分析結果（常駐地點的語意類別）
- **時序規律性特徵**：個人軌跡穩定性分析的量化指標

### 7.3 模型輸入格式

- 每筆訓練樣本為單一使用者某一天的完整軌跡序列，長度為 48（對應 48 個時階段）。
- 每個時間步的特徵包含網格座標 (x, y) 及對應的 POI 特徵。

## 8. 評估指標

| 指標 | 說明 |
|------|------|
| **GEO-BLEU** | 計算預測軌跡與實際軌跡的空間序列相似性，參考 NLP 中的 BLEU 指標設計，適用於地理軌跡的 n-gram 比較。 |
| **FDE（Final Displacement Error）** | 預測軌跡最後一個時間點的位置與真實位置之間的距離誤差，衡量長期預測的終點精度。 |

---

## 9. 預計成果目標

| 目標 | 說明 |
|------|------|
| **準確率** | 軌跡預測準確率超越資料集所提供的 baseline（以 GEO-BLEU 與 FDE 衡量）。 |
| **作息分析** | 依照各時段的軌跡分析使用者的日常作息規律，產出可解讀的人流行為報告。 |
| **可解釋化** | 能解釋模型如何利用空間、語意、規律性條件預測使用者的軌跡，提升模型透明度。 |

## 11. 專案結構

```
final-project/
├── raw_data/
│   ├── nagoya_challengedata.csv      # 主要資料集（名古屋，約 1.11 億筆）
│   ├── sapporo_challengedata.csv
│   ├── hiroshima_challengedata.csv
│   └── kumamoto_challengedata.csv
│
├── data/
│   ├── loader.py        # 讀取 CSV、切 train(d≤60) / test(d>60)
│   └── preprocessing.py # 時間處理（48時階段、K-means平假日）、空間處理
│
├── analysis/
│   ├── eda.py           # 探索性分析（人流分布、每日趨勢、熱力圖）
│   ├── clustering.py    # HDBSCAN（識別移動熱點）
│   └── trajectory.py    # 個人軌跡穩定性分析、分群繪圖
│
├── models/
│   ├── baseline.py      # Per-User Mode、Bigram 等 baseline
│   └── cvae.py          # Conditional VAE 模型定義與訓練
│
├── eval/
│   └── metrics.py       # 封裝 GEO-BLEU（Beta=0.5, n=5）、FDE，產出評估報告
│
├── docs/
│   └── proposal.md
└── main.py              # 串接各模組的執行入口
```

### 模組依賴關係

```text
loader → preprocessing → clustering ─┐
                       → trajectory  ─┼→ cvae → metrics
                       → baseline ───┘
                       → eda（獨立，隨時可跑）
```

### 開發優先順序

| 優先序 | 模組 | 說明 |
|--------|------|------|
| 1 | `data/` | 其他模組的共同基礎，需先完成 |
| 2 | `eval/metrics.py` | 儘早確認評估流程可跑通 |
| 2 | `models/baseline.py` | 取得本地 GEO-BLEU 基準分數 |
| 2 | `analysis/eda.py` | 獨立模組，可平行進行 |
| 3 | `analysis/clustering.py` | 需 loader |
| 3 | `analysis/trajectory.py` | 需 loader |
| 4 | `models/cvae.py` | 需等 clustering + trajectory 完成 |
